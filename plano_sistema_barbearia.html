Objetivo
Sistema SaaS de agendamento para barbearias
Multi-tenant. Cada barbearia é um cliente com seu próprio subdomínio, painel e agenda. Você hospeda, eles pagam mensalidade.
Stack back-end
Java 21 Spring Boot 3 Spring Security JWT JPA / Hibernate PostgreSQL Redis Docker Railway / Render
Stack front-end
React 18 TypeScript Vite React Query React Hook Form Tailwind CSS shadcn/ui date-fns Vercel

Multi-tenant
Cada Tenant tem seu próprio slug. Todo endpoint filtra automaticamente por tenantId via interceptor. Um banco único, dados isolados por coluna.
Autenticação e segurança
Spring Security com JWT stateless. Dois fluxos: painel admin (dono/Funcionario) usa Bearer token. Página pública de agendamento não precisa de auth — só valida o slug da barbearia.
JWT + Refresh Token BCrypt senha Role: OWNER / BARBER CORS configurado
Deploy back-end
Railway.app — grátis no início, simples de configurar, suporta PostgreSQL e Redis nativamente. Basta conectar o repositório GitHub e fazer push.

Como funciona o controle de 30 dias gratuito
Simples: você armazena a data de criação do tenant no banco. Um campo trialExpiresAt calculado automaticamente. Um interceptor Spring verifica antes de cada request admin.
Fluxo técnico do trial
Tenant criado → trialExpiresAt = now + 30d
Request admin chega
→ Interceptor lê tenant do JWT
→ Verifica: isActive || !isTrialExpired()
→ Se expirado: retorna 402 Payment Required
→ Front exibe banner "Período trial encerrado"
Página pública sempre funciona
Mesmo com trial expirado, a página de agendamento do cliente final continua acessível. Isso é intencional — cria pressão: o dono recebe agendamentos mas não consegue ver o painel. Incentivo forte para pagar.
