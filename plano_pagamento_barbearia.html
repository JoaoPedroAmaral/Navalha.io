Objetivo
Sistema SaaS de agendamento para barbearias
Multi-tenant. Cada barbearia tem seu próprio subdomínio, painel e agenda. Você hospeda, eles pagam mensalidade. Cobrança automatizada via Asaas.

Fluxo completo — trial até pagamento recorrente
1
Cadastro da barbearia
Você cadastra o tenant no sistema. Automaticamente: trialExpiresAt = hoje + 30 dias, isActive = true. Você também cria o cliente no Asaas e salva o asaasCustomerId.
2
30 dias de trial livre
Dono usa tudo normalmente. No dia 25 você envia mensagem no WhatsApp avisando que o trial acaba em 5 dias. Custo: zero para os dois.
3
Trial expira
O interceptor Spring detecta trialExpiresAt no passado e isActive = false. Retorna 402 para rotas admin. A página pública de agendamento continua funcionando — pressão natural para pagar.
4
Dono vê tela de pagamento
Front exibe tela bloqueante com as opções: Pix, boleto ou cartão. Ao clicar, o front chama POST /billing/charge no seu back-end.
5
Back-end gera cobrança no Asaas
Seu servidor chama a API do Asaas com o asaasCustomerId, valor R$ 99 e tipo de pagamento. O Asaas retorna o QR code Pix ou link de boleto. Você devolve pro front exibir.
6
Pagamento confirmado — webhook
Quando o Pix é pago (ou boleto compensado), o Asaas chama automaticamente POST /webhook/asaas no seu back-end. Você valida a assinatura, seta isActive = true e nextBillingDate = hoje + 30 dias.
7
Painel reativado automaticamente
Na próxima requisição do front, o interceptor libera o acesso. Sem intervenção manual sua. No mês seguinte, o ciclo se repete.
Nenhum dado de cartão passa pelo seu servidor
Cartão é processado direto pelo Asaas. Você só recebe o evento "pagou". Zero responsabilidade com PCI compliance.

boolean trialValid = tenant.getTrialExpiresAt()

Job de verificação diária
Um @Scheduled roda toda meia-noite verificando tenants com nextBillingDate = hoje. Gera automaticamente a cobrança do mês seguinte no Asaas e manda notificação pro dono.
Inadimplência
Se o pagamento não vier em 3 dias após o vencimento, seta isActive = false. O painel bloqueia automaticamente. Quando pagar, o webhook reativa.
Tela de pagamento (bloqueante)
Quando a API retorna 402, o front substitui o painel por uma tela de pagamento. Não é um modal — é uma tela completa que não tem como fechar sem pagar.
Botão Pix (preferido) Botão Boleto Botão Cartão QR Code na tela Polling de confirmação
Polling de confirmação (Pix)
Após gerar o QR code Pix, o front faz polling a cada 5 segundos em GET /billing/status. Quando o back-end confirmar pagamento via webhook, o próximo poll retorna { "active": true } e o front redireciona pro painel automaticamente — experiência fluida, sem precisar recarregar a página.
Banner de aviso (dias 25–30 do trial)
Antes de bloquear, o painel exibe um banner amarelo discreto: "Seu período gratuito encerra em X dias. Garanta a continuidade agora." O front calcula isso comparando a data local com trialExpiresAt retornado no JWT ou num endpoint GET /admin/billing/status.
